## 1）“HY-Embodied-0.5中使用的ViT是HY-ViT的升级版本”到底是什么意思？

这句话的直接意思是：

**HY-Embodied-0.5 没有从零重新发明一个全新的视觉编码器，而是在他们原来已有的一条视觉模型路线 HY-ViT 的基础上，做了升级，形成了 HY-ViT 2.0，并把它接到这个具身模型里。** 论文原文同时说它是 “HY-ViT series 的 advanced iteration（进阶版本）”，并且强调它保留了“原生支持任意分辨率输入”的能力。 

更具体地说，“升级版本”通常包含三层含义：

1. **继承同一模型家族的设计路线**
   也就是它们不是完全不相关的两个模型，而是前后代关系。

2. **保留旧版的核心优点**
   这里保留的是“native-resolution / arbitrary-resolution input”，即原生支持任意分辨率输入。

3. **在训练数据、监督方式、部署效率上做增强**
   论文明确说 HY-ViT 2.0 用了更大规模预训练数据、引入 tiny LLM 的语言监督、加入视觉重建监督，并且为了边缘部署选择了 400M 参数版本。

所以这句话不是在说“这就是一个行业通用标准模型”，而是在说：**这是腾讯混元自己那条视觉编码器路线的新版。**

## 3）什么叫“引入一个 tiny LLM 来提供语言监督信号”？

这个地方论文写得比较短，**没有把具体训练公式展开**，所以这里我要分成“明确内容”和“合理解释”两层。

### 论文明确说了什么

论文明确写的是：HY-ViT 2.0 “introduces a tiny LLM to provide language supervision signals”，也就是**训练视觉编码器时，引入一个小型语言模型，让视觉特征受到语言侧监督**。

### 这在技术上通常是什么意思

它的核心意思不是“拿一个小模型来替代大模型”，而是：

**视觉编码器输出的特征，不仅要对图像本身有用，还要对“后面的语言模型”有用。**

也就是让视觉特征朝着“可被语言模型理解、可和文字语义对齐”的方向去学。

你可以把这理解成下面这个训练目标：

* 不是只让 ViT 学“看见边缘、纹理、目标类别”；
* 还让 ViT 学“这些视觉特征以后如何被语言系统解释成概念、关系和描述”。

举个直觉例子：

图里有“桌子上的红杯子”。
如果没有语言监督，ViT 可能学到的是：

* 一个红色区域
* 一个圆柱形边缘
* 一个器皿状物体

但有语言监督时，它会更倾向于把这些视觉特征组织成更适合语言接收的语义表示，比如：

* “红色杯子”
* “在桌子上”
* “可作为可交互对象”

所以 tiny LLM 的作用，通常可以理解为：

**在视觉编码阶段提前注入“语言可解释性”与“跨模态对齐压力”。**

### 为什么要用 tiny LLM，而不是直接拿大 LLM 监督？

因为训练一个完整大 LLM 参与视觉监督，代价太高。
tiny LLM 可以提供一个比较轻量的语言语义约束，让视觉特征更像“语言能接收的 embedding”，但成本比直接拉满大模型低很多。

### 这句话最本质的含义

一句话总结就是：

**他们不想让 ViT 只学“视觉识别器”，而是想让它学成“给 VLM/LLM 喂语义的视觉前端”。**
这是“语言监督信号”的本质。论文明确说 ViT 的目标是把视觉输入投影到语言嵌入空间中，从而让 LLM 无缝处理视觉和文本。

## 4）什么叫“视觉重建监督（visual reconstruction supervision）”？

这个相对好理解。

它的核心意思是：

**训练视觉编码器时，不仅要求它提取高层语义，还要求它保留足够多的原始视觉信息，使这些特征能够支持“把图像内容还原出来”或者至少还原出紧凑的视觉表示。** 论文原文说这样做是为了“确保喂给 LLM 的视觉信号的信息损失尽可能小”。

为什么这很重要？

因为纯语义训练容易让 ViT 只保留“是什么”，丢掉“长什么样、在哪里、细节如何”。

比如：

* 只做分类监督，模型知道这是“杯子”
* 但可能丢掉杯把手朝左、杯口直径、图中文字、精细边缘、空间位置这些细节

而具身任务、空间任务、OCR、精细感知都非常依赖这些细节。

所以“视觉重建监督”本质是在逼迫视觉特征满足两件事：

1. **语义上有用**
2. **信息上不塌缩**

也就是：
不仅要“知道图里是什么”，还要“尽量别把图里细节压没了”。

在这篇论文后面，作者又说他们训练了一个更大的 ViT 去生成离散视觉表示，并用这个离散表示监督视觉 token；2.2 节还写到 vision branch 会去预测下一个 patch 的 discrete code。这个其实就很像一种“重建/预测式监督”的具体落地。

## 5）为什么“400M ViT + 从更强内部 ViT 蒸馏”能兼顾高效和准确？

这里可以拆成两部分：**为什么高效**，以及**为什么准确**。

### 为什么高效

因为边缘设备最怕的是：

* 参数太多
* 计算太大
* 延迟太高
* 显存/内存占用太高

一个 400M 参数的 ViT，虽然不小，但相比更大的视觉骨干，已经更适合部署。论文明确说他们就是为了保证边缘设备实时性，才采用 400M 参数 ViT。

所以“高效”的来源首先是：

**学生模型本身被做得足够小。**

不是蒸馏让它更快，而是**小模型天然更快**。

### 为什么还能保持准确

因为他们不是直接训练这个小 ViT，而是让它向一个更强的内部大 ViT 学习，也就是做蒸馏。论文原文说：通过从更强大的内部 ViT 蒸馏训练，帮助模型获得高效且准确的视觉表示。

蒸馏的直觉是：

* 大模型老师学得更好，视觉表征更强；
* 小模型学生容量更小，但可以模仿老师的输出分布、中间特征或目标表示；
* 这样学生会比“自己从头硬学”更容易逼近高质量表征。

你可以把它理解成：

> 不是让一个 400M 模型自己慢慢摸索“什么是好视觉特征”，
> 而是让它直接朝着一个已经很强的大老师的表征去拟合。

所以组合起来就是：

* **小模型本身** 带来部署速度
* **蒸馏训练** 带来性能保底

这就是为什么它能同时兼顾“边缘实时性”和“表示准确度”。

### 但要注意一点

这不是说“只要蒸馏就一定实时”。
更准确地说是：

**他们先选了适合部署的 400M 架构，再用蒸馏尽量把大模型性能灌进去。**

## 6）为什么还要再训练一个“更大规模的 ViT”去生成离散视觉表示？

这个设计非常关键。它说明作者其实做了两件事：

1. 一个 **部署用的小 ViT（400M）**
2. 一个 **训练时当老师/视觉 tokenizer 用的更大 ViT**

论文说这个更大的 ViT 用来生成一种“既支持理解也支持重建”的离散视觉表示，然后把这种表示拿来监督模型的视觉 token。

为什么要这么做？

因为直接拿原始像素做监督，往往不适合给 LLM 分支或视觉 token 提供高质量训练信号。原始像素太底层、太密、太难学。
而如果先由一个更强的大 ViT把图像压成“高质量视觉 token 序列”，再让小模型去学这个序列，就会更稳定。

它相当于在做一件事：

**先把图像翻译成一种离散的“视觉语言”，再让下游模型去学这套视觉语言。**

为什么这个离散表示要求“既支持理解又支持重建”？

因为如果它只支持理解，那可能太语义化，很多细节会丢失；
如果它只支持重建，那又可能太像低层压缩码，语义不够强。

所以作者想要的是一种折中表示：

* 一方面保留视觉细节，便于重建；
* 一方面又抽取了较高层结构，便于理解。

这样它就很适合做监督目标。

## 7）什么叫 codebook？

你可以把 **codebook** 理解成一个“有限大小的视觉词典”或“离散原型库”。

论文说它的 codebook size 是 **2k**，也就是这个视觉词典里有大约 **2048 个离散条目**。

每个条目都可以理解成一个“视觉原型向量”：

* 有的代表某类边缘纹理
* 有的代表某种局部形状
* 有的代表某种颜色+结构模式
* 有的代表更复杂的小块视觉模式

当图像的某个局部 patch 进来时，模型不会直接保留一串连续浮点数，而是会说：

> 这个 patch 最像 codebook 里的第 137 号原型
> 或者第 892 号原型

于是，这个 patch 就被表示成一个**离散 ID**。

这和 VQ-VAE、视觉 tokenizer、离散视觉 token 的思路非常接近。

## 8）什么叫“每个 8×8 图像 patch 被压缩为一个离散 code”？

这句话可以直接按流程理解：

### 第一步：把图像切成很多 8×8 小块

假设一张图像被分解成许多局部 patch，每个 patch 大小是 8×8 像素。论文明确写了“compresses every 8×8 image patch into a single discrete code”。

### 第二步：每个 patch 不再用原始像素表示

原本一个 8×8 patch 如果是 RGB，就有：

[
8 \times 8 \times 3 = 192
]

个原始数值。

现在不直接保留这 192 个像素值，而是通过大 ViT/视觉 tokenizer，把这个 patch 映射成一个 codebook 里的编号。

比如：

* patch A → code 57
* patch B → code 1302
* patch C → code 411

于是，一整张图就能变成像这样的一串离散 token：

[
[z_1, z_2, z_3, \dots, z_N], \quad z_i \in {1,\dots,2048}
]

### 第三步：为什么说这是“压缩”？

因为原来每个 patch 是很多连续像素值，
现在每个 patch 只用**一个离散 ID**表示。

这当然会有信息损失，但如果 codebook 学得好，这个 ID 会尽量保留最关键的局部视觉模式。

所以“8×8 patch 压成一个离散 code”的本质就是：

**把连续图像，转成离散视觉 token 序列。**

## 9）这个离散 code 在他们模型里具体拿来干什么？

这个在 2.2 节说得更清楚。论文写到：

* 使用大 ViT 生成的 discrete visual representations 作为监督；
* 在 vision branch 上加一个 MLP；
* 让模型去预测 **下一个 patch 的 discrete code**；
* 这对应 Figure 2 里的 **Vision Loss**。

所以它不是“仅仅为了压缩”，而是为了给视觉分支一个非常强的训练目标：

**让视觉 token 不只是服务最终文本输出，而是自己也学会一种类似“视觉语言建模”的任务。**

有点像 NLP 里预测下一个 token，只不过这里预测的是下一个视觉 patch 的离散 code。

这会带来两个好处：

1. **视觉分支训练信号更强**
   不用只靠最后的 LLM loss 反传。

2. **细粒度视觉建模更强**
   因为 patch-level 离散 code 会逼模型关注局部结构和视觉细节。

## 1）“引入一个 tiny LLM 来提供语言监督信号”，实际操作时怎么做？

### 1.1 先说它的目标

这句话的核心不是“再塞一个小语言模型进去推理”，而是：

**训练视觉编码器时，不让它只学视觉统计模式，还让它输出一种“语言模型容易读懂”的视觉表示。**
因为论文明确说 ViT 的作用是把视觉输入投影到语言 embedding 空间，让 LLM 能无缝处理视觉和文本。

---

### 1.2 最可能的实际训练方式

论文没有给出 tiny LLM 的具体损失公式，所以这里我只能做**高可信还原**。最常见也最符合这篇话术的做法是下面这样：

#### 方案 A：把 tiny LLM 当一个轻量文本解码器

1. 输入图像 `I`
2. 用 ViT 得到视觉 token：
   [
   v_1, v_2, \dots, v_N
   ]
3. 通过一个 projector，把这些视觉 token 映射到语言模型 hidden size
4. 把这些映射后的视觉 token 喂给一个 tiny LLM
5. 让 tiny LLM 去预测和图像对应的文字，比如：

   * caption
   * object tags
   * region description
   * QA answer
6. 对 tiny LLM 的文本输出做标准交叉熵损失

也就是：

[
\mathcal{L}*{lang_sup} = - \sum_t \log p(y_t \mid y*{<t}, v_1,\dots,v_N)
]

这里的反向传播会穿过 tiny LLM，继续传回 ViT，于是 ViT 会被逼着学会：
**“输出的视觉特征要足够适合语言生成。”**

这就是“语言监督信号”的真正含义。它不是一句抽象口号，而是：
**用文本任务的损失，反过来约束视觉表示。**
这和论文说的“用 tiny LLM 提供 language supervision signals”以及“把视觉输入投到语言嵌入空间”是一致的。

---

### 1.3 你可以把它想成一个两阶段接口训练

如果写成非常直观的伪代码，大概像这样：

```python
image = load_image()
visual_tokens = vit(image)                 # [N, d_v]
visual_tokens = projector(visual_tokens)   # [N, d_lm]

# tiny LLM 读视觉 token，然后预测配套文本
loss_lang = tiny_llm_loss(visual_tokens, target_text)

loss_lang.backward()
```

这里 tiny LLM 的角色不是“最终模型的一部分一定要上线部署”，而更像一个**训练期老师 / 约束器**。
它告诉 ViT：你的输出不能只是“视觉上像样”，还必须“语言上可读”。

---

### 1.4 为什么要 tiny，而不是直接拿大 LLM 来做这件事？

原因很直接：

* 大 LLM 太贵
* 训练视觉编码器时，没必要每步都拉上超大语言模型
* tiny LLM 足够提供“语言侧梯度方向”

所以它像一个便宜很多的“语言判官”。
它不负责最终上限，但负责把视觉表征往“可语言化”的方向拽。这个设计和论文强调的边缘效率取向是一致的。

---

## 2）什么叫“离散视觉表示”？

“离散视觉表示”对应的是：
**不用连续浮点向量直接表示一个视觉 patch，而是把它量化成一个有限词表里的编号。**
论文明确说：他们训练了一个更大的 ViT 来生成 discrete visual representations，这个表示有一个大小为 2k 的 codebook，并把每个 `8×8` 图像 patch 压成一个 discrete code。

---

### 2.1 连续表示 vs 离散表示

普通 ViT 对一个 patch 的输出通常是连续向量：

[
h_i \in \mathbb{R}^d
]

比如 1024 维、2048 维，里面每个数都是 float。

而离散表示不是输出一个 float 向量，而是输出一个**编号**：

[
z_i \in {1, 2, \dots, K}
]

这里 (K=2048)（也就是论文说的 2k codebook）。
于是一个 patch 不再是“很多连续值”，而是“词表里的第 731 号视觉单元”。

---

### 2.2 什么叫 codebook？

**codebook 就是一个视觉词典。**

你可以把它理解成一个表：

[
\mathcal{C} = {c_1, c_2, \dots, c_{2048}}
]

每个 (c_k) 都是一个“视觉原型向量”。
当某个 patch 进来时，系统会找：

> 这个 patch 最像词典里的哪一个原型？

如果最像第 137 个原型，那它就被编码成离散编号 `137`。

所以 codebook 本质上就是：

**把无限多的局部视觉模式，压到有限个可索引原型上。** 

---

### 2.3 什么叫“每个 8×8 图像 patch 被压缩为一个离散 code”？

比如一张 RGB 图像里，一个 `8×8` patch 原始有：

[
8 \times 8 \times 3 = 192
]

个像素值。
现在不直接保留这 192 个值，而是让一个大 ViT/视觉 tokenizer 说：

> 这个 patch 最像 codebook 里的第 842 类

于是这个 patch 被表示成一个整数 `842`。

整张图就能变成：

[
[z_1, z_2, \dots, z_N], \quad z_i \in {1,\dots,2048}
]

这就叫“把图像变成离散视觉 token 序列”。论文原文就是这么描述的。

---

## 3）什么叫“把这种表示拿来监督模型的视觉 token”？

这句话在论文里其实已经给出了非常关键的实现信息：
他们说使用更大 ViT 产生的离散视觉表示作为监督，在视觉分支的 LLM 输出特征上加一个 MLP，去预测**下一个 patch 的 discrete code**；3.3 节又给出了对应的 vision loss 是对视觉 token 做交叉熵。

这件事可以拆成下面几步。

---

### 3.1 老师先产出“标准答案”

更大的 ViT 先把图像切成 patch，并为每个 patch 产出离散 code：

[
z_1, z_2, \dots, z_N
]

这些 (z_i) 就是监督目标，也就是“老师答案”。

---

### 3.2 学生模型也会得到视觉 token

HY-Embodied-0.5 的视觉分支在经过 ViT + MoT 后，会得到一串视觉 hidden states：

[
h_1, h_2, \dots, h_N
]

这些 (h_i) 还是连续向量，不是离散编号。

---

### 3.3 用一个 MLP 把连续向量映射到“2048类分类”

对每个 (h_i)，接一个 MLP 分类头：

[
p_i = \text{Softmax}(\text{MLP}(h_i))
]

其中 (p_i \in \mathbb{R}^{2048})，表示这个位置上各个 code 的概率。
然后和老师给的离散目标 (z_i) 做交叉熵：

[
L_{vision} = - \frac{1}{N_v} \sum_i \log p_i(z_i)
]

这就是论文 3.3 节明确写出的 vision loss 形式。

---

### 3.4 为什么这叫“监督视觉 token”？

因为 loss 不是直接打在像素上，也不是直接打在文本输出上，
而是打在**视觉分支产生的 hidden states / visual tokens** 上。

也就是说：

* 视觉 token 如果学得不好
* 就预测不准老师给的离散视觉 code
* 于是交叉熵变大
* 梯度回传，逼视觉 token 变得更“像能解释 patch 的表示”

所以“拿离散视觉表示监督视觉 token”的意思就是：

**让视觉 token 学会预测一个由强老师生成的 patch-level 视觉标签序列。** 

---

## 4）什么叫 VQ-VAE、视觉 tokenizer、离散视觉 token？

这三个词本质上是同一类思想的不同说法。

### 4.1 VQ-VAE

VQ-VAE 的核心思想就是：

1. 编码器把输入映射到连续向量
2. 再把这个连续向量“量化”到最近的 codebook entry
3. 得到离散编号
4. 解码器再尝试从这些离散编号重建输入

所以它的关键词就是：
**vector quantization（向量量化）+ codebook + 离散 token**。

---

### 4.2 视觉 tokenizer

视觉 tokenizer 就是把图像像文字分词一样，切成“视觉 token”。

文字 tokenizer 做的是：

> “我爱机器学习” → [“我”, “爱”, “机器”, “学习”]

视觉 tokenizer 做的是：

> 图像 patch → [842, 137, 55, 901, ...]

所以它本质是：
**把连续图像离散化成 token 序列。**

---

### 4.3 离散视觉 token

就是视觉 tokenizer 产出的那些离散 ID。
比如 `842`、`137`、`55` 这种编号，就是离散视觉 token。

所以这三个术语你可以这么记：

* **VQ-VAE**：一种实现离散视觉编码的经典框架
* **视觉 tokenizer**：这类模块的功能名称
* **离散视觉 token**：最终产出的离散编号结果

这和论文里说的 “discrete visual representations / codebook / discrete code” 是同一条技术路线。

---

## 5）什么叫“自适应计算架构”？

自适应计算架构的意思是：

**不同输入，不一定走完全相同的参数和计算路径。**

普通 Transformer 是“所有 token 走同一套层、同一套参数”。
自适应计算则是“根据 token 类型 / 内容 / 难度，走不同专家、不同分支、不同参数”。

论文明确说，这类架构常见实现包括 MoE 和 MoT，它们被用来在计算效率和性能之间取得平衡。

---

## 6）什么叫 MoE 和 MoT？

### 6.1 MoE（Mixture-of-Experts）

MoE 一般是：

* 一层里有很多 expert
* 每个 token 经过 router
* router 选择少数几个 expert 来处理这个 token

也就是：

[
y = \sum_{e \in \text{selected experts}} g_e(x), E_e(x)
]

其中 (g_e(x)) 是路由权重，(E_e) 是专家网络。

特点是：

* 总参数量很大
* 每次只激活少数专家
* 计算比“全都跑一遍”省

---

### 6.2 MoT（Mixture-of-Transformers）

MoT 的核心也是“不是所有 token 共用一套 Transformer 参数”，但它不一定像 MoE 那样通过 router 按 token 动态选专家。

在这篇论文里，MoT 更接近：

**按模态划分分支：视觉 token 走视觉版 QKV/FFN，文本 token 走文本版 QKV/FFN。**
图 2 也明确画出了 `Vision MoT` 和 `Language MoT` 两条分支，并且说明 MoT 通过 modality-specific QKV 和 FFN 来解耦视觉与文本 token 的处理。

所以你可以把这篇里的 MoT 理解成：

> 不是“一个 token 去好多专家里挑几个”，
> 而是“视觉 token 和文本 token 直接进不同 Transformer 子参数组”。

---

## 7）什么叫“为语言 token 和视觉 token 引入非共享参数”？

这个最关键。

### 7.1 共享参数是什么意思？

普通多模态 LLM 通常是这样的：

* 图像先变成视觉 token
* 文本 token 和视觉 token 拼成一个长序列
* 然后一起过同一个 Transformer
* 也就是同一套 `QKV`、同一套 `FFN`

这叫**共享参数**。

---

### 7.2 非共享参数是什么意思？

这篇论文做的是：

* 文本 token 还是走原来 LLM 的那套参数
* 视觉 token 改走另一套复制出来的参数

也就是每一层里，至少这两类参数变成两套：

* 文本版：
  [
  W_Q^{txt}, W_K^{txt}, W_V^{txt}, W_{FFN}^{txt}
  ]
* 视觉版：
  [
  W_Q^{vis}, W_K^{vis}, W_V^{vis}, W_{FFN}^{vis}
  ]

这就叫**non-shared parameters**。论文原文也是这样说的。

---

## 8）为什么这样能“提升视觉建模能力，同时缓解语言能力退化”？

因为**共享参数会发生梯度冲突**。

### 8.1 如果共享一套参数，会发生什么？

文本 token 需要的能力是：

* 语言建模
* 语法
* 世界知识
* 推理链生成

视觉 token 需要的能力是：

* 局部纹理
* 边缘
* patch 关系
* 空间结构
* 多视角细节

这两类分布差异非常大。

如果你让它们都去更新同一套 QKV/FFN，就会出现：

* 为了更好处理视觉 token，参数往“视觉统计规律”偏
* 但这会破坏原来预训练 LLM 擅长的语言行为

这就是论文说的 “heavy visual training 导致 inherent language capabilities degradation”。

---

### 8.2 非共享参数为什么能缓解？

因为现在变成：

* 文本 token 更新文本参数
* 视觉 token 更新视觉参数

于是：

* 视觉分支可以放心朝视觉最优方向适配
* 文本分支则尽量保住原来 LLM 的语言能力

所以“视觉强 + 语言不太掉”就是参数解耦带来的。
这不是玄学，是典型的“减少模态间参数干扰”。论文也明确把这一点作为 MoT 的动机。

---

## 9）什么叫“在多模态训练开始之前”？

这句话是在说一个**时间点**：

> 先有一个已经预训练好的纯文本 LLM，
> 然后在“开始图文联合训练之前”，把里面的一些参数复制出来，作为视觉分支初值。

也就是顺序大概是：

1. 先有 pretrained LLM
2. 把其中每层的 QKV、FFN 拷贝一份
3. 原来的保留给文本 token 用
4. 拷贝出的那份作为视觉 token 的专用参数
5. 然后才开始真正的 multimodal training

这就是论文里说的 “before multi-modal training begins”。

---

### 9.1 为什么会有“训练前 / 训练后”的区分？

因为这篇模型不是一口气从随机初始化训练到底。
论文明确区分了：

* **large-scale pre-training**
* **mid-training**
* **fine-tuning / post-training**

而且在不同阶段，损失和可训练模块都不完全一样：
预训练时有 `Lllm + Lvision + Lglobal`；到后续 mid-training 和 fine-tuning，又把 vision/global 监督去掉，只保留标准语言损失。

所以“before multi-modal training begins”就是在**从纯文本 LLM 切到多模态训练**之前做结构改造。

---

## 10）什么叫“复制 FFN 和 QKV 参数，并用预训练 LLM 权重初始化这组复制参数”？

这句话可以直接翻译成代码层面。

假设某一层原始 LLM 有这些参数：

* Attention 部分
  [
  W_Q, W_K, W_V
  ]
* FFN 部分
  [
  W_1, W_2
  ]
  或者门控 MLP 的
  [
  W_{up}, W_{gate}, W_{down}
  ]

那他们做的事情就是：

[
W_Q^{vis} \leftarrow W_Q^{txt},\quad
W_K^{vis} \leftarrow W_K^{txt},\quad
W_V^{vis} \leftarrow W_V^{txt}
]

[
W_{FFN}^{vis} \leftarrow W_{FFN}^{txt}
]

也就是为视觉分支新建一套参数，并把初始值设成文本分支那套预训练权重。
论文说得很明确：复制 FFN 和 QKV，并用 pre-trained LLM 的权重初始化。

---

### 10.1 复制到哪里去？

复制到每一层里的**视觉专用子模块**里，也就是图 2 里的 `Vision MoT` 分支。
文本原参数留在 `Language MoT` 分支。图 2 也明确画出了视觉和语言两套 QKV/FFN。

---

### 10.2 为什么复制后能直接用？

因为此时视觉 token 已经被 ViT + projector 映射到了和语言模型一致的 hidden size。
论文明确说 ViT 会把视觉输入投影到语言 embedding space。既然 token 维度已经对齐，那视觉 token 在张量形状上就可以走和文本 token 相同结构的 Transformer 层。

也就是说，复制出来的参数不是“硬凑”，而是因为：

* 输入维度相同
* 层结构相同
* 所以可以直接套进同样的 QKV/FFN 公式

只是处理对象从文字 embedding 变成了视觉 embedding。

---

## 11）什么叫“视觉 token 用复制参数计算，文本 token 用原始参数计算”？

这句话说的是**同一层内部的 token 路由规则**。

假设一层输入是一个交错序列：

[
[x_1, x_2, \dots, x_T]
]

其中有的 token 是视觉，有的是文本。
那么这一层不会让所有 token 都统一过同一套 QKV/FFN，而是：

* 如果 (x_i) 是视觉 token：

  * 用视觉版参数
    [
    W_Q^{vis}, W_K^{vis}, W_V^{vis}, W_{FFN}^{vis}
    ]
* 如果 (x_i) 是文本 token：

  * 用文本版参数
    [
    W_Q^{txt}, W_K^{txt}, W_V^{txt}, W_{FFN}^{txt}
    ]

这就是论文说的 “all visual tokens ... use duplicated parameters, while text tokens use original text-specific parameters”。

---

### 11.1 如果写成伪代码，大概就是这样

```python
for layer in transformer_layers:
    for token in sequence:
        if token.is_visual:
            q = token @ Wq_vis[layer]
            k = token @ Wk_vis[layer]
            v = token @ Wv_vis[layer]
            token = attention_vis(q, k, v, mask=visual_mask)
            token = ffn_vis[layer](token)
        else:
            q = token @ Wq_txt[layer]
            k = token @ Wk_txt[layer]
            v = token @ Wv_txt[layer]
            token = attention_txt(q, k, v, mask=text_mask)
            token = ffn_txt[layer](token)
```

这不是论文原始代码，但和论文描述、图 2 / 图 3 的含义是一致的。 

---

### 11.2 注意：这不代表视觉和文本完全隔离

论文图 2 和 2.3 说明，他们还加入了：

* visual latent token
* mixed optimization loss
* 不同 attention 机制

所以视觉和文本不是“永远互不通信”，而是：

**它们在底层参数上做了解耦，但在整体序列建模和训练目标上仍然被桥接起来。**
这也是论文说 visual latent token 能连接视觉与文本内容的原因。 

---

## 12）这整套流程，实际跑起来是什么样？

你可以把它理解成下面这条完整数据流。

### 第一步：图像进 ViT

图像 `I` 经过 HY-ViT 2.0，得到视觉 token：

[
v_1,\dots,v_N
]

并且这些 token 已经被映射到语言 embedding 空间。

### 第二步：加 visual latent token

每个视觉元素末尾再加一个 learnable latent token。

### 第三步：和文本 token 拼接

形成类似：

[
[\text{img tokens}, \text{latent}, \text{text prompt}, \text{target text}]
]

的序列，图 3 也是按这种交错多模态序列画的。

### 第四步：进入每一层 MoT

同一层里：

* 视觉 token 走 vision-QKV / vision-FFN
* 文本 token 走 text-QKV / text-FFN
* attention mask 也不同：视觉更适合双向 attention，文本是 causal attention。

### 第五步：训练时同时吃三类损失

在预训练阶段：

* 文本 answer 用 `Lllm`
* 视觉 token 做 next-code prediction，用 `Lvision`
* latent token 对齐老师全局特征，用 `Lglobal`

总损失：

[
L_{total}=L_{llm}+L_{vision}+L_{global}
]

这是论文明确给出的。

---

## 13）一句话压缩总结

你刚才问的这一大串，其实都指向同一件事：

**他们不是把图像 token 和文本 token 粗暴拼起来就完了，而是先把视觉表示做成“语言可读 + 细节不丢”的形式，再在 Transformer 内部给视觉 token 单独开一条参数分支，让视觉能力增强，同时尽量不破坏原有 LLM 的语言能力。**
这就是 tiny LLM 监督、离散视觉表示、MoT 非共享参数、vision loss 这些设计背后的统一逻辑。

* 单张图像：([v_1, v_2, \dots, v_N, \ell])
* 两张图像：([v^{(1)}*1,\dots,v^{(1)}*{N_1}, \ell^{(1)}, v^{(2)}*1,\dots,v^{(2)}*{N_2}, \ell^{(2)}])

这里 (\ell) 才是那个 **visual latent token**。论文原文就是 “append a learnable visual latent token to the end of each visual element (e.g., an image or a video frame)”；同时 Figure 2 也把它单独画成位于 `Visual Inputs` 和 `Text Inputs` 之间的一小段 `Visual Latent`。

---

## 一、Figure 2 到底在画什么？

这张图可以从**下往上**看。

### 1. 最底下：输入层

底部有三种输入：

* **Visual Inputs**：由图像经过 `HY-ViT 2.0-400M` 编码后得到的视觉 token
* **Visual Latent**：每个视觉元素末尾附加的一个可学习 latent token
* **Text Inputs**：文本提示词、问题、指令等 token

图左下角的 `HY-ViT 2.0-400M` 表示：图像先进入视觉编码器，再变成可送入上面 MoT 主干的视觉 token。论文明确说 ViT 的作用是把视觉输入投到语言 embedding 空间，从而让 LLM 能统一处理视觉和文本。

---

### 2. 中间：MoT 主干

中间主体是 `HY-Embodied-0.5 MoT-2B`，里面分成左右两块：

* 左边是 **Vision MoT**
* 右边是 **Language MoT**

每边都各自有：

* 一套 **QKV**
* 一套 **FFN**

这表示视觉 token 和文本 token **不共用同一套 QKV/FFN 参数**。
论文和图注都明确说，MoT 用 modality-specific 的 QKV 和 FFN，把视觉 token 和文本 token 的处理解耦。

---

### 3. 中间那条注意力层

图里中间横着写了两种注意力：

* 左边：**Local Full Attn**
* 右边：**Global Causal Attn**

它表达的核心不是“完全两个模型不通信”，而是：

* 视觉侧更偏向**双向/全注意力**
* 文本侧保留**因果注意力**
* 两种模态在同一个整体序列里协同，但各自的计算方式不同

论文正文也说了：视觉数据不像语言那样有天然单向性，因此**双向 attention 更适合视觉建模**；文本依然保持自回归语言模型的 causal attention。

---

### 4. 最上面：三个训练目标

顶部有三种 loss：

* **Vision Loss**：监督视觉 token
* **Global Loss**：监督 visual latent token
* **LLM Loss**：监督文本输出 `<think> ... <answer> ...`

论文第 3.3 节明确说，这三个 loss 分别监督 visual tokens、latent tokens、language tokens；预训练阶段联合优化
[
L_{total}=L_{llm}+L_{vision}+L_{global}
]
而到后续 mid-training / fine-tuning，就去掉 vision/global 监督，只保留标准语言损失。

---

## 二、什么叫“可学习的视觉潜在向量 / token”？

你可以把它先理解成：

**插在一整张图像 token 序列末尾的一个“可训练摘要槽位”**。

它不是图像里真实切出来的 patch，也不是某个像素块。
它更像一个额外引入的参数向量，初始时是可学习参数，和其他视觉 token 一起进入 Transformer，在层层 self-attention 里不断吸收整张图的信息。论文还明确说，这些 latent visual tokens 在训练时本身也是可训练参数。

更形式化一点：

如果图像经过 ViT 得到
[
v_1,\dots,v_N
]
那么加了 latent token 后序列变成
[
[v_1,\dots,v_N,\ell]
]

这里 (\ell) 就是 visual latent token。
它的意义不是“表示某一个 patch”，而是希望经过多层交互后，它变成一个**汇聚整张图全局语义/全局状态**的向量。

---

## 三、它不是“每个 token 后面加一个”，而是“每个视觉元素后面加一个”

这点非常重要。

### 错误理解

[
[v_1,\ell_1,v_2,\ell_2,v_3,\ell_3,\dots]
]

这不是论文的意思。

### 正确理解

单图像：
[
[v_1,v_2,\dots,v_N,\ell]
]

双图像：
[
[v^{(1)}*1,\dots,v^{(1)}*{N_1},\ell^{(1)},v^{(2)}*1,\dots,v^{(2)}*{N_2},\ell^{(2)}]
]

视频也是一样：**每一帧末尾一个 latent token**。
这和论文原文 “the end of each visual element” 完全一致。

---

## 四、这个 latent token 在推理时起什么作用？

推理时它**不是被拿来直接输出答案的特殊头**，而是作为序列中的一个中间状态，继续参与后续 Transformer 计算。

更直观地说，它在推理时有三个作用。

### 1. 充当“全局摘要位”

普通 patch token 各自偏局部。
latent token 因为位置特殊、目标特殊，会更倾向于吸收整张图像的全局信息，比如：

* 场景整体状态
* 关键物体之间的关系
* 当前任务相关的全局语义

也就是它更像“这一张图的 summary token”。

---

### 2. 充当视觉到语言的接口

因为文本 token 后面还要继续读整个多模态序列。
如果视觉侧只有大量 patch token，语言侧要从一堆局部 token 里自己拼全局含义，成本很高。latent token 相当于额外提供了一个**已经被压缩/汇聚过的视觉摘要位**，让后面的语言部分更容易读取高层语义。论文直接说它被设计来 “connect vision and language”，并且图注也说 visual latent tokens 用来 bridge 模态关系。

---

### 3. 充当训练时被重点塑形的“桥接节点”

它不是随便多加一个 token，而是**被单独施加 Global Loss**。
所以训练后，它会比普通 patch token 更强地承载“整图语义 + 跨模态对齐”的职责。

---

## 五、什么叫“用一个大规模 ViT 的全局特征来监督该 token 的输出特征”？

这是最核心的一句。

论文原文意思是：

* 先有一个更强的 teacher ViT
* teacher ViT 能提取整张图的全局特征，文中说的是它的 **global CLS feature**
* 然后拿当前模型里 visual latent token 的输出特征，去对齐这个 teacher 的全局特征
* 这个对齐损失就是 **Global Loss**。

论文第 3.3 节甚至把公式写出来了：

[
L_{global} = - \frac{f^\top_{latent} f_{teacher}}
{|f_{latent}||f_{teacher}|}
]

也就是**负余弦相似度**。其中：

* (f_{latent})：latent token 输出后再做映射得到的向量
* (f_{teacher})：teacher ViT 提取的全局 CLS 特征 

---

### 这句话翻成大白话就是：

他们在逼这个 latent token 学会：

> “你不要只做一个普通附加 token，
> 你要尽量长成一个能代表整张图全局语义的向量。”

也就是说，**teacher ViT 的全局图像表示，是 latent token 的老师答案**。

---

## 六、为什么这叫“监督该 token 的输出特征”？

因为监督信号不是直接打在原始输入 token 上，而是打在**这个 latent token 经过主干网络计算后的最终 hidden state** 上。

也就是：

1. 先把 latent token 和视觉 token 一起送进网络
2. 经过多层 attention / FFN 之后，它得到最终表示
3. 再把这个最终表示拿去和 teacher 的全局特征对齐

所以监督的是：

**这个 token 学完之后“变成了什么样”**，而不是它初始参数是什么。

---

## 七、为什么说它能够“连接视觉内容和文本内容”？

这个地方分成两层：

### 1. 结构层面的原因

它刚好位于视觉序列和文本序列之间。Figure 2 里它就画在 `Visual Inputs` 和 `Text Inputs` 中间。
而且它本身来自视觉侧，却又不是普通局部 patch token，而是被训练成全局语义摘要。这使它天然适合作为视觉侧向语言侧传递高层语义的“接口位”。

---

### 2. 训练层面的原因

它被单独用 Global Loss 拉向 teacher 的全局图像语义；同时整个模型还有 LLM Loss，要求最终生成文本回答。
于是这个 token 同时受到两种压力：

* 一边要忠实表达整张图的视觉全局语义
* 一边又要服务后续语言生成

所以它自然会学成一种“既懂图，又便于语言读取”的中间表示。

---

### 3. 作者给出的实证证据

论文后面在 Fig.12 的可视化分析里说，visual latent tokens 的视觉注意力会落到关键物体、关键部件和重要空间区域上，而语言注意力会集中在核心语义实体、状态描述、空间关系和动作指令上；作者据此认为 latent token 的确在桥接视觉细节与语言概念。这个结论是作者给出的观察，不是我额外脑补。

---

## 八、你可以把它类比成什么？

最像的类比有两个。

### 类比 1：ViT 里的 CLS token

CLS token 会汇聚整张图信息，用来做分类。
这里的 visual latent token 和 CLS 有点像，但用途更偏**跨模态桥接**，不是只做单模态分类。

### 类比 2：给每张图额外放一个“总结位”

普通 patch token 像是细节记录员；
latent token 像是这张图的“总负责人”，负责把全局有用信息浓缩出来，交给后面的语言部分。

---

## 九、把 Figure 2 整体串起来

你可以把这张图理解成下面这条信息流：

### 第一步：图像编码

图像先经过 `HY-ViT 2.0-400M`，变成视觉 token。

### 第二步：追加视觉 latent token

每张图像 / 每帧视频的 token 序列末尾，再加一个可学习 latent token。

### 第三步：与文本 token 一起进入 MoT

进入主干后：

* 视觉 token 走 Vision MoT 的 QKV/FFN
* 文本 token 走 Language MoT 的 QKV/FFN
* 注意力机制也分视觉侧和语言侧。

### 第四步：三种损失同时训练

* **Vision Loss**：让视觉 token 预测 teacher ViT 产生的离散 patch code
* **Global Loss**：让 latent token 对齐 teacher ViT 的全局 CLS 特征
* **LLM Loss**：让文本输出正确的 `<think> ... <answer> ...`。

所以你会发现：

**普通视觉 token 负责“细节感知”**，
**visual latent token 负责“全局摘要 + 跨模态桥接”**，
**文本 token 负责“推理与生成”**。

这就是 Figure 2 最核心的逻辑。

## 十、最简洁的结论

你这里最关键的三个点，可以压缩成这三句：

1. **visual latent token 不是每个 patch 后面一个，而是每张图/每帧末尾一个。** 
2. **它在训练时被 Global Loss 专门塑造成“整图全局语义摘要位”。** 
3. **它之所以能连接视觉和文本，是因为它一头吸收视觉全局信息，一头服务文本生成，因此学成了跨模态接口。**

下一条我可以直接给你画一个**Figure 2 的“张量形状版”**，比如：
[
I \to [v_1,\dots,v_N,\ell] \to \text{MoT layers} \to \text{text decoding}
]
把每一步 tensor shape、loss 位置、latent token 的更新过程全写出来。
Q:请展开讲解训练策略这一部分，这三种损失函数分别是怎么发挥作用的，为什么预训练阶段要这样联合优化，为什么后续阶段仅保留标准的自回归语言损失？
A:可以，这一部分其实是整篇 paper 里最关键的地方之一。作者不是只用一个统一的 loss 去“糊”整个模型，而是把监督拆成了三层：

[
L_{\text{total}}=L_{\text{llm}}+L_{\text{vision}}+L_{\text{global}}
]

并且明确让这三项分别去监督 **语言 token、普通视觉 token、visual latent token**。作者在 3.3 节就是这么定义的；Figure 2 的图注也强调，visual latent tokens 和 mixed optimization losses 是用来在大规模训练中“桥接并强化”模态之间关系的。

## 一、先从整体看：这三种 loss 分别在“管谁”

可以把整条前向链路想成这样：

图像先经过 HY-ViT 2.0，变成一串视觉 token；每个视觉元素末尾再加一个 visual latent token；然后它们和文本 token 一起进入 MoT 主干。此时模型里其实有三类“被学习的表示”：

1. **普通视觉 token**：负责局部 patch、细粒度视觉模式
2. **visual latent token**：负责整张图的全局语义摘要
3. **语言 token**：负责推理链和最终文本输出

所以三种 loss 的本质分工是：

* **(L_{\text{vision}})**：把“局部视觉表征”学扎实
* **(L_{\text{global}})**：把“全局图像摘要”学扎实
* **(L_{\text{llm}})**：把“面向任务的理解、推理、生成”学扎实 

也就是说，作者是在把“看细节”“抓全局”“说出来”这三件事拆开监督。

---

## 二、(L_{\text{vision}}) 到底在做什么

论文定义的 vision loss 是：

[
L_{\text{vision}}=-\frac{1}{N_v}\sum_{i=1}^{N_v}\log p_i(z_i)
]

其中：

* (N_v) 是视觉 token 数量
* (p_i) 是第 (i) 个视觉 token 对离散 code 的预测分布
* (z_i) 是 teacher ViT 生成的目标离散 code。

这件事的真实含义是：

**让模型里的视觉分支，学会预测“下一个 patch 应该对应哪个离散视觉 code”。**

也就是说，它不是直接拿像素去监督，也不是只靠最后答案对不对来反推视觉表示，而是给视觉分支一个非常明确的 patch-level 学习目标。论文在 2.2 节也明确说了，引入 visual next-code prediction 是因为多模态数据中超过一半是视觉 token，他们希望用更强的监督来更好地优化 MoT 中的视觉分支。

### 这个 loss 的作用，你可以理解成三层

第一层，它让视觉 token 学会**细粒度局部结构**。
因为监督单位是 patch 对应的离散 code，所以模型必须关心边缘、纹理、局部形状、空间邻接关系，不然就预测不准。

第二层，它让视觉分支获得**直接监督**。
如果没有这个 loss，视觉分支只能通过最后的文本回答误差间接学习。那条梯度链太长，而且文本答案通常只约束高层语义，不足以稳定塑造 patch-level 表征。

第三层，它让视觉表示更像一种**可建模的“视觉语言”**。
因为 target 不是连续像素，而是大 ViT 产出的离散 code，相当于 teacher 先把图像翻译成了一串高质量视觉 token，再让学生去模仿。这样监督比原始像素更抽象、更稳定，也更贴近后续 Transformer 擅长处理的 token 形式。

所以一句话概括：

**(L_{\text{vision}}) 负责把模型的“看图能力”做厚，尤其是局部、细粒度、可组合的视觉能力。**

---

## 三、(L_{\text{global}}) 到底在做什么

论文把 global loss 定义为 latent token 映射后的隐藏状态 (f_{\text{latent}}) 与 teacher ViT 的全局 CLS 特征 (f_{\text{teacher}}) 之间的**负余弦相似度**：

[
L_{\text{global}}=-\frac{f_{\text{latent}}^\top f_{\text{teacher}}}{|f_{\text{latent}}||f_{\text{teacher}}|}
]

也就是逼两者方向尽量一致。

这意味着作者在强行要求 visual latent token 去学一件事：

> 你要长成“这一整张图的全局语义摘要”。

因为 teacher ViT 的 CLS 特征本身就是全局图像表示，所以把 latent token 去对齐它，本质就是在把 latent token 塑造成一个“整图 summary slot”。论文在 2.3 节也明确说，预训练阶段他们用大 ViT 的 global features 去监督这个 token，并观察到它能有效连接视觉内容和文本内容。

### 为什么这里不用交叉熵，而用 cosine similarity

因为这里的目标不是做分类，不是让 latent token 选出“第几类 code”，而是让它对齐一个**连续的全局语义向量**。
这种目标更关心“表示方向是否一致”，而不是某个离散类别是否命中，所以用余弦相似度更自然。

### 这个 loss 的核心作用

它解决的是一个和 (L_{\text{vision}}) 不一样的问题：

* (L_{\text{vision}}) 学的是“局部 patch 怎么表示”
* (L_{\text{global}}) 学的是“整张图怎么浓缩成一个高层摘要”

如果没有这项 loss，latent token 虽然也可能通过自注意力吸到一些信息，但它不一定会稳定成长为“全局语义接口”。有了这项专门监督之后，它就被塑造成一个更适合后续语言部分读取的跨模态桥。

所以一句话概括：

**(L_{\text{global}}) 负责把模型的“整图概括能力”和“视觉到语言的接口位”做稳。**

---

## 四、(L_{\text{llm}}) 到底在做什么

这一项其实是最熟悉的标准自回归语言损失。论文在 3.3 节说得很清楚：它监督 language tokens；在后续 mid-training 和 fine-tuning 阶段，也是只保留这一个。

但这里有一个容易误解的点：

**只叫“语言损失”，不代表它只训练语言模块。**

因为在多模态 VLM 里，文本输出是建立在“图像 token + latent token + 文本 prompt”共同输入的基础上的。所以当答案 token 出错时，梯度仍然会穿过整个多模态计算路径回传。只不过它监督的位置在**输出文本**，不是在 patch 级视觉表示或全局视觉摘要上。

### 这项 loss 主要负责三件事

第一，它训练模型把视觉输入转成**可回答的语义**。
也就是从“看到了什么”进一步变成“能说出来什么”。

第二，它训练模型学会**推理链与回答格式**。
Figure 2 里顶部就画了 `<think> ... <answer> ...`，说明作者希望模型不只是识别，还要生成 reasoning trace 和最终答案。

第三，它把视觉表示真正纳入到**下游任务目标**里。
前两个 loss 更多是“表征塑形”，而 (L_{\text{llm}}) 是“任务落地”。

所以一句话概括：

**(L_{\text{llm}}) 负责把“看懂”变成“会推理、会作答、会执行任务接口”。**

---

## 五、为什么预训练阶段要把三种 loss 联合优化

这其实是这篇文章最聪明的地方之一。

如果预训练时只用 (L_{\text{llm}})，模型当然也能学，但会有两个明显问题：

第一，**视觉监督太间接**。
答案级别的语言 loss 对视觉分支来说太远了。它会告诉你“这个问题答错了”，但不会告诉你“是哪个 patch 的边缘、深度、局部结构没学好”。

第二，**全局摘要位不一定能自然长出来**。
latent token 如果没有专门的 global loss，它只是一个额外 token，不保证一定会变成稳定的全局语义接口。

所以作者在预训练阶段同时保留三种目标，本质上是在同时学三件事：

* 用 (L_{\text{vision}}) 学**局部细节**
* 用 (L_{\text{global}}) 学**全局语义**
* 用 (L_{\text{llm}}) 学**跨模态生成与推理**

这三个目标一起上，等于把视觉表征空间从两个方向夹住：

* 从下往上：patch 级视觉结构不能塌
* 从上往下：整图语义摘要不能散
* 最终还要落到文本任务上：这些表示必须对回答有用

这就能在训练最早期把模型的视觉端“立起来”。而论文在第 3 节开头也明确说，他们的策略是在最早期训练阶段就嵌入物理世界理解能力；并在大规模预训练阶段引入 2D/3D grounding、深度估计、分割等多样视觉感知数据，以增强对复杂物理环境的感知与解释能力。

你可以把联合优化理解成一句话：

**预训练阶段不是只教模型“回答正确”，而是同时教它“怎么看局部、怎么抓全局、怎么把看到的东西说出来”。**

---

## 六、为什么后续阶段只保留标准自回归语言损失

论文明确写了：在后续 mid-training 和所有 fine-tuning 阶段，他们丢弃 vision 和 global supervision，只优化标准的 autoregressive language loss；而在 mid-training 阶段又进一步冻结了全部 ViT 参数，只更新 HY-Embodied-0.5 模块。

这背后的逻辑，可以分成“论文明确做法”和“为什么这么合理”两层来看。

### 1. 论文明确做法

后续阶段的目标已经从“建立基础视觉表征”转向“对齐具身任务和推理能力”。
mid-training 里他们引入的是更高质量、更复杂的 embodied / spatial / general understanding 数据，并统一 prompt 模板和坐标格式；同时不再训练 ViT，只更新后续模块。

### 2. 为什么这时要去掉 (L_{\text{vision}}) 和 (L_{\text{global}})

因为这两个 loss 更像是**基础表征塑形目标**，而不是最终任务目标。

当预训练已经把视觉端塑得比较稳定以后，后面更重要的是：

* 指令跟随
* 长短链推理
* 具身问答
* 空间 reasoning
* 任务规划输出

这些目标本质上都体现在**文本生成质量**上。
所以这时候只保留 (L_{\text{llm}})，能让优化焦点集中到“怎样把已有视觉表征更好地用于任务回答”。

换句话说：

* 前期：重点是“把眼睛练好”
* 后期：重点是“学会用这双眼睛完成任务”

### 3. 如果后期还一直保留 vision/global loss，会怎样

虽然论文没有直接写“会有坏处”，但从训练逻辑上很容易理解：

如果后期仍然强推视觉 token 去预测 teacher 的离散 code、强推 latent token 去对齐 teacher 的全局 CLS，那么模型就会持续被拉回“做视觉表征拟合”这个目标。
而后续阶段真正需要的是围绕具身任务做**推理、决策、回答格式对齐**。这两类目标未必完全一致。

所以去掉这两个辅助视觉目标，相当于告诉模型：

> 基础视觉能力先别乱动太多了，
> 现在主要练如何把这些视觉信息用于下游任务生成。

这和他们在 mid-training 冻结 ViT 是完全一致的：
**视觉底座定型，后面重点练上层任务能力。** 

---

## 七、把这套训练策略压缩成一句非常本质的话

这篇 paper 的训练策略，本质上是在分两个阶段做两件不同的事：

### 预训练阶段

同时优化
[
L_{\text{llm}}+L_{\text{vision}}+L_{\text{global}}
]
目的不是单纯提高答案准确率，而是**同时把局部视觉、全局视觉、语言生成三种能力一起打底**。

### 后续阶段

只保留
[
L_{\text{llm}}
]
目的不是再继续雕刻底层视觉表征，而是**把已经学好的视觉能力，转化成具身任务上的理解、推理、规划和输出能力**。

---

## 八、你可以把三种 loss 记成这三个角色

* **(L_{\text{vision}})**：教模型“看清局部细节”
* **(L_{\text{global}})**：教模型“抓住整图语义”
* **(L_{\text{llm}})**：教模型“把看到的东西用于推理和回答”

而整个训练 schedule 的核心思想就是：

**先把视觉底座和跨模态接口练稳，再把优化重点切到任务级生成。**

# **4 后训练（Post-training）**

---

## **4.1 监督微调（Supervised Fine-tuning, SFT）**

### **4.1.1 数据构建（Data Construction）**

在监督微调（SFT）阶段，我们重点强化模型的**长链推理能力**。为此，我们从前述的空间、具身以及通用数据源中筛选出一部分**高复杂度、多步骤问题**，并结合额外的内部推理数据。

对于这些样本，我们通过**人类与模型协作的流程**构建 **Chain-of-Thought（CoT）推理轨迹**。随后，这些生成的 CoT 会由一个 LLM 从多个维度进行评估，包括：

* 推理质量
* 逻辑正确性
* 序列重复情况

同时，我们还对最终答案进行**精确匹配验证**。

通过这一流程，我们构建了约 **10万（100k）条冷启动 CoT 数据**，用于训练 MoT-2B 和 MoE-A32B 模型。 

---

### **4.1.2 训练流程（Training Recipe）**

在冷启动 SFT 阶段，我们仍然使用标准的**交叉熵损失**进行优化。

但与预训练和中期训练不同的是：

* **关闭 sequence packing（序列打包）**
* 每个样本单独处理

这样做的目的是：
**强调每条推理链的独立性**，避免不同样本之间的干扰。

训练过程中：

* 学习率保持为 **5e-5** 

---

## **4.2 强化学习（Reinforcement Learning）**

---

### **4.2.1 数据构建（Data Construction）**

在强化学习阶段，我们**动态构建训练数据**，而不是使用固定数据集。

具体方法：

1. 维护一个大规模候选样本池（覆盖各种具身能力）
2. 在每一轮 RL 中，用当前模型对这些样本进行多次采样评估
3. 根据结果筛选数据：

* 全部成功 → 太简单 → 丢弃
* 全部失败 → 太难 → 丢弃
* 部分成功 → 保留（关键样本）

这些“部分成功”的样本位于模型当前能力边界附近，提供最有价值的学习信号。

此外，为避免模型过拟合某一类能力，我们还在以下维度上做平衡：

* 感知
* 预测
* 交互
* 规划

每一轮 RL 使用 **5万（50K）样本**，并随着模型能力提升持续更新数据分布，从而形成一种**能力自适应课程学习（curriculum learning）**。 

---

### **4.2.2 奖励设计（Reward Design）**

具身任务的一个核心挑战是：**输出形式高度多样**，包括：

* 几何定位
* 轨迹预测
* 离散决策
* 连续估计
* 开放式推理

因此无法用一个统一 reward。

我们采用**任务感知（task-aware）奖励设计**：

[
r = R_t(y, y^\star) \in [0,1]
]

核心原则：

* 有结构的任务 → 用确定性奖励
* 无法精确判断的任务 → 用 LLM judge

---

### **四类奖励函数**

#### **1. Grounding-based（定位类）**

用于空间定位：

* IoU
* Hungarian IoU
* 点距离
* Chamfer 距离

👉 提供**连续梯度式反馈**

---

#### **2. Regression-based（数值类）**

用于：

* 计数
* 距离估计

使用**相对误差衰减函数**

---

#### **3. Trajectory-based（轨迹类）**

用于：

* 路径规划
* 动作序列

使用：

* DTW
* Fréchet 距离
* 终点一致性

👉 能区分“部分正确”和“完全错误”

---

#### **4. Textual-based（文本类）**

* 固定答案 → 精确匹配
* 开放回答 → LLM Judge

[
r_{\text{free}} = J(q, y, y^\star)
]

---

### **核心思想**

👉 **奖励结构必须匹配输出结构**

* 连续问题 → 连续 reward
* 离散问题 → exact match
* 开放问题 → LLM judge

这种混合设计对稳定 RL 非常关键。 

---

### **4.2.3 训练流程（Training Recipe）**

使用 **GRPO（Group Relative Policy Optimization）**：

对于输入 (x)，采样 (G=16) 个回答：

[
A_i = \frac{r_i - \mu(r)}{\sigma(r)}
]

👉 使用组内归一化 advantage（解决 reward 尺度不一致问题）

---

优化目标：

[
\mathcal{L}_{RL} = - \sum \min(\rho A, \text{clip}(\rho) A)
]

其中：

[
\rho = \frac{\pi_\theta}{\pi_{\theta_{\text{old}}}}
]

---

### **训练细节**

* group size = 16
* batch size = 128
* 学习率 = (8\times10^{-7})
* 训练 5 epoch
* 最大上下文 = 16k
* 最大输出 = 16k

额外稳定措施：

* 去掉 reward variance = 0 的样本
* 限制过长输出
* 非对称 clipping：[0.8, 1.35]



---

## **4.3 迭代式深度推理训练（RFT）**

RL 能提高 reward，但**不能保证推理过程质量**。

因此引入 **RFT（Rejection Sampling Fine-Tuning）**：

流程：

1. 用当前模型生成多个答案
2. 筛选：

   * 全对 → 丢弃
   * 全错 → 丢弃
   * 部分正确 → 保留
3. 用强 teacher 评估推理质量
4. 只保留高质量推理轨迹

最终：

* 100万候选 → 30万高质量数据

---

### **RFT vs RL 的关系**

* RL：探索能力边界
* RFT：固化优质推理路径

👉 RL 负责“发现”，RFT 负责“记住”

---

### **整体策略**

交替进行：

[
\text{RL} \rightarrow \text{RFT} \rightarrow \text{RL} \rightarrow ...
]

👉 将“偶然成功”变成“稳定能力”



---

## **4.4 大到小的 On-Policy 蒸馏（OPD）**

目标：
把大模型能力迁移到小模型（不是简单压缩）。

---

### **关键问题**

传统蒸馏的问题：

* 只学 teacher 的输出
* 不考虑 student 自己的错误轨迹

---

### **OPD 的核心思想**

1. student 自己生成：

[
y \sim \pi_s
]

2. teacher 在 student 的轨迹上做 teacher forcing

3. 对齐两者分布：

[
\mathcal{L}_{OPD} = \sum KL(\pi_t || \pi_s)
]

---

### **为什么更好**

👉 student 在“自己会犯错的地方”被监督

👉 减少 train-test mismatch

👉 学到完整推理过程，而不是只学答案

---

### **整个后训练 pipeline**

* RL：扩展能力边界
* RFT：提炼优质推理
* OPD：迁移到小模型

👉 最终实现：

**大模型发现能力 → 小模型继承能力**



---

# ✅ 一句话总结第4章

这套后训练流程本质是在做三件事：

* **SFT：教模型怎么推理（基础能力）**
* **RL + RFT：让模型学会更强、更稳定的推理**
* **OPD：把这些能力压缩迁移到小模型**
